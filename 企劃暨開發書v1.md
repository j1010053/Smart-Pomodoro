# Smart Pomodoro 企劃暨開發書 v1

## 1. 產品定位

Smart Pomodoro 是一個以手機使用為主的個人工作輔助工具。它不是單純番茄鐘，也不是完整專案管理系統，而是用來降低日常工作前阻力、協助使用者把任務丟進系統、快速進入工作，並透過長期資料逐步優化自己的工作流。

核心目標：

```txt
低阻力輸入任務
整理今日可行工作
快速進入番茄工作循環
記錄執行結果
分析任務、時間、重要性、緊急性與完成度之間的關係
逐步校正分類預設、推薦排序與工作流配置
```

v0.1 採用 PWA 本機優先架構。核心功能不依賴伺服器，不依賴 AI，不需要帳號登入。AI 可作為橫跨功能組的可選模組，未來透過後端服務接入。

## 2. 功能組總覽

產品分為五個功能組：

```txt
1. 前期規劃組
   以最小阻力幫助使用者把工作丟進系統。

2. 工作組
   以番茄鐘為核心，協助使用者快速進入工作並維持工作品質。

3. 優化組
   根據長期資料分析任務分類、時間安排、矩陣象限與完成結果的關係。

4. AI 組
   橫跨規劃、工作、優化三組，但不作為 v0.1 核心依賴。

5. 系統組
   負責本機儲存、備份、還原、PWA、離線支援、資料版本管理。
```

## 3. 前期規劃組

### 3.1 功能敘述

前期規劃組的目標是降低輸入成本。使用者不應一開始就被要求做完整專案管理，而是可以先把腦中的任務快速丟進系統，再由系統用分類預設參數補齊初始資訊。

MVP 功能：

```txt
快速卸載流程
單任務輸入
預設工作分類
任務分類預設參數
重要程度設定
期限設定
能量需求設定
預估時間設定
平均工作量設定
```

### 3.2 快速卸載流程

流程：

```txt
1. 使用者輸入多行文字，一行一件事。
2. 系統拆成多個暫存任務。
3. 使用者為每個任務快速確認分類、重要程度、期限、大小、能量。
4. 系統依分類預設補齊必要參數。
5. 任務寫入本機資料庫。
```

快速標記問題應保持輕量：

```txt
這件事屬於哪一類？
這件事重要嗎？
它有時間壓力嗎？
做起來費力嗎？
大概多大？
```

避免在前期規劃階段要求完整專案結構、精確工時、複雜標籤、甘特圖或帳號登入。

### 3.3 任務分類預設參數

任務分類不是固定真理，而是可被優化組校正的輸入假設。使用者選擇分類後，系統帶出預設重要程度、能量需求、預估時間、建議時段與矩陣目標占比。

建議預設分類：

```txt
深度工作
行政雜務
例行任務
學習成長
溝通協作
生活事項
```

資料模型：

```ts
type TaskTypeProfile = {
  id: string;
  name: string;
  defaultImportance: 1 | 2 | 3 | 4;
  defaultEnergy: 1 | 2 | 3;
  defaultEstimateMinutes: number;
  preferredTimeWindow?: "morning" | "afternoon" | "evening" | "any";
  targetShare?: number;
  active: boolean;
};
```

使用者可以接受、拒絕或手動修改優化組提出的分類預設調整。

### 3.4 任務資料模型

`priority` 應明確定義為「重要程度」，參照艾森豪矩陣的 Important 軸，不再混用為緊急程度。若重構時可改名，建議改成 `importance`。

```ts
type Task = {
  id: string;
  title: string;
  typeId: string;
  importance: 1 | 2 | 3 | 4;
  deadline?: string;
  energy: 1 | 2 | 3;
  estimateMinutes: number;
  doneMinutes: number;
  repeatDays?: number[];
  active: boolean;
  createdAt: string;
  updatedAt: string;
};
```

緊急程度不建議由使用者手動輸入，應主要由 deadline 自動計算。

### 3.5 平均工作量設定

```ts
type WorkSettings = {
  dailyWorkMinutes: number;
  averageEnergy: 1 | 2 | 3;
  bufferRatio: number;
};
```

有效每日容量：

```txt
effectiveDailyCapacity = dailyWorkMinutes * (1 - bufferRatio)
```

預設 `bufferRatio = 0.15`，代表保留 15% 緩衝。

## 4. 工作組

### 4.1 功能敘述

工作組以番茄鐘為核心，目標是讓使用者打開 App 後快速知道現在該做什麼，並能立即開始。

MVP 功能：

```txt
Focus 首頁
番茄鐘
5 分鐘啟動模式
推薦任務序列
跳過警告
今日容量提示
完成一輪後累加任務進度
推薦原因顯示
```

首頁應突出單一主行動：

```txt
現在建議做：任務名稱
開始 25 分鐘
```

輔助操作：

```txt
換一個
跳過
只做 5 分鐘
完成一輪
```

### 4.2 番茄鐘實作邏輯

手機背景執行不可靠，因此計時器不應只靠 `setInterval` 每秒遞減，而應使用時間戳計算剩餘時間。

資料模型：

```ts
type TimerMode = "focus" | "shortBreak" | "longBreak" | "microStart";

type TimerSession = {
  id: string;
  taskId?: string;
  mode: TimerMode;
  plannedSeconds: number;
  startedAt?: string;
  plannedEndAt?: string;
  endedAt?: string;
  pausedRemainingSeconds?: number;
  completed: boolean;
};
```

剩餘時間：

```txt
remainingSeconds = plannedEndAt - now
```

切到背景再回來時，依目前時間重新計算剩餘時間。

### 4.3 艾森豪矩陣

推薦系統需同時處理「重要」與「緊急」。

```txt
重要 = importance
緊急 = deadlineUrgency(deadline)
```

四象限：

```txt
第一象限：重要且緊急
第二象限：重要但不緊急
第三象限：不重要但緊急
第四象限：不重要且不緊急
```

緊急程度算法：

```ts
function urgencyFromDeadline(deadline?: string): number {
  if (!deadline) return 0;

  const days = daysUntil(deadline);

  if (days < 0) return 100;
  if (days === 0) return 80;
  if (days <= 1) return 60;
  if (days <= 3) return 35;
  if (days <= 7) return 15;
  return 5;
}
```

分類：

```ts
function isImportant(task: Task) {
  return task.importance >= 3;
}

function isUrgent(task: Task) {
  return urgencyFromDeadline(task.deadline) >= 35;
}
```

### 4.4 推薦排序算法

推薦不應只是單任務分數，而應是情境化工作流排序。

輸入：

```ts
type RecommendationContext = {
  now: string;
  hour: number;
  weekday: number;
  currentEnergy: 1 | 2 | 3;
  remainingDailyCapacity: number;
  completedToday: string[];
  previousTaskId?: string;
  previousTaskTypeId?: string;
  todaySequence: string[];
};
```

總分：

```txt
finalScore =
  baseTaskScore
  + timeFitScore
  + relationScore
  + sequenceScore
  + matrixBalanceScore
  - conflictPenalty
  - overloadPenalty
```

基礎分數：

```txt
baseTaskScore =
  urgencyScore
  + importanceScore
  + quadrantBias
  + energyFit
  + progressMomentum
  - recentCompletionPenalty
```

建議權重：

```txt
urgencyScore = urgencyFromDeadline(deadline)
importanceScore = importance * 12

quadrantBias:
  重要且緊急 = +25
  重要不緊急 = +18
  不重要但緊急 = -5
  不重要不緊急 = -20

energyFit:
  當前能量符合任務需求 = +6
  當前能量低於任務需求 = -8

progressMomentum:
  已開始但未完成 = +5

recentCompletionPenalty:
  今天已做過同任務，每次 -6
```

第二象限需有保護分數，避免長期重要任務被短期緊急任務持續擠掉。

### 4.5 關係描述插入推薦排序

任務模型保持乾淨，不新增特殊任務屬性。與時間、整體工作流、任務相鄰關係有關的效果，應以「關係描述」形式插入推薦排序。

資料模型：

```ts
type WorkflowRelation = {
  id: string;
  source: RelationSource;
  target: RelationTarget;
  relationType:
    | "promotes"
    | "suppresses"
    | "conflictsWith"
    | "prefersTime"
    | "prefersPosition";
  condition?: RelationCondition;
  effect: {
    metric:
      | "completionRate"
      | "importantTaskCompletion"
      | "dailyThroughput"
      | "skipRate"
      | "focusQuality";
    direction: "increase" | "decrease";
    effectSize: number;
    confidence: number;
  };
  enabled: boolean;
};
```

可描述的情況：

```txt
某分類在上午完成率較高
某分類靠近另一分類時會互相抑制
某分類排在第一輪時，當天總完成度提高
高能量任務放在低能量時段時跳過率上升
溝通類任務靠近深度工作時，深度工作完成率下降
```

關係分數：

```txt
relationScore = effectSize * confidence * relationWeight
```

抑制關係：

```txt
conflictPenalty = abs(effectSize) * confidence * penaltyWeight
```

促進關係：

```txt
promotionBonus = effectSize * confidence * bonusWeight
```

關係來源：

```ts
type RelationSource =
  | "systemDefault"
  | "userConfigured"
  | "analyticsDetected"
  | "aiSuggested";
```

優先級：

```txt
使用者設定 > 長期分析偵測 > 系統預設 > AI 建議
```

推薦卡片需要顯示原因，例如：

```txt
重要程度高
今天尚未處理重要不緊急任務
這類任務在上午完成率較高
上一個任務後接它的跳過率較低
```

使用者應可關閉或降低某條規則權重。

### 4.6 推薦序列

v0.1 可先採用貪婪法：

```txt
1. 選出目前最高分任務。
2. 更新 context。
3. 根據上一個任務重新計算下一輪分數。
4. 產生今日前 3 到 5 個推薦任務。
```

v0.2 可升級為 beam search：

```txt
保留前 N 條候選序列
計算任務總分、相鄰關係分數、時段適配、矩陣平衡與容量限制
選出最高分序列
```

## 5. 優化組

### 5.1 功能敘述

優化組不是單純回顧完成了多少，而是分析任務、時間、能量、重要緊急矩陣與完成結果之間的促進與抑制關係。

目標：

```txt
找出哪些安排讓工作更容易開始
找出哪些緊急事項侵占重要事項
找出哪些低重要任務其實與整體完成度正相關
監測任務分類是否有效
協助使用者逐步接近艾森豪矩陣理想配比
```

注意：不應加入「暖機」這類特殊任務參數。這類現象應由關係分析描述，而不是污染任務資料模型。

### 5.2 事件紀錄

優化組依賴事件資料。v0.1 應先把事件記好。

```ts
type WorkEvent = {
  id: string;
  type:
    | "task_created"
    | "task_started"
    | "task_completed"
    | "task_skipped"
    | "timer_started"
    | "timer_completed"
    | "task_rescheduled"
    | "task_reclassified"
    | "settings_updated";
  taskId?: string;
  timestamp: string;
  context: {
    hour: number;
    weekday: number;
    energyAtStart?: 1 | 2 | 3;
    dailyCapacityMinutes: number;
    completedBeforeToday: number;
    quadrant: EisenhowerQuadrant;
    taskTypeId: string;
    importance: 1 | 2 | 3 | 4;
    urgencyScore: number;
  };
};
```

### 5.3 象限統計

```ts
type QuadrantStats = {
  quadrant:
    | "importantUrgent"
    | "importantNotUrgent"
    | "notImportantUrgent"
    | "notImportantNotUrgent";
  taskCount: number;
  completedCount: number;
  skippedCount: number;
  completedMinutes: number;
  plannedMinutes: number;
  completionRate: number;
  skipRate: number;
};
```

分析問題：

```txt
重要不緊急任務是否長期完成率偏低？
不重要但緊急任務是否佔用太多時間？
重要且緊急任務是否常由拖延造成？
高重要任務是否跳過率過高？
低重要任務是否反而完成很多？
```

### 5.4 詰抗指數

詰抗指數不只是四象限比例失衡，而是用來觀察某類任務、某種安排、某個時段是否促進或抑制其他工作結果。

基礎指標：

```txt
重要流失率：
  重要任務被跳過或延期的比例

緊急侵占率：
  不重要但緊急任務佔用總完成時間的比例

第二象限缺席率：
  最近 7 天沒有完成任何重要不緊急任務的天數比例
```

計算：

```txt
importantLossRate =
  skippedImportantTasks / allImportantTaskAttempts

urgentCrowdingRate =
  completedMinutesOfNotImportantUrgent / totalCompletedMinutes

q2AbsenceRate =
  daysWithoutImportantNotUrgentWork / observedDays

eisenhowerFriction =
  importantLossRate * 0.4
  + urgentCrowdingRate * 0.35
  + q2AbsenceRate * 0.25
```

解讀：

```txt
0.0 - 0.3：健康
0.3 - 0.6：緊急事項開始壓迫重要事項
0.6 - 1.0：長期重要工作被系統性犧牲
```

### 5.5 促進與抑制關係偵測

優化組需偵測任務分類、象限、時段與相鄰任務之間的關係。

例：

```txt
如果某分類任務出現在一天前兩輪，
當天總完成番茄數是否提高？

如果某分類任務完成後接高重要任務，
高重要任務完成率是否下降？

如果低重要低緊急任務在特定位置出現，
是否提高當日總完成度？
```

效果值：

```txt
promotionEffect =
  avgCompletionDaysWithPattern - avgCompletionDaysWithoutPattern

suppressionEffect =
  avgImportantCompletionWithoutPattern - avgImportantCompletionWithPattern
```

若樣本數不足，關係不得自動啟用，只能顯示為低信心觀察。

### 5.6 任務分類有效性監測

分類有效性是指分類是否能預測工作結果。

監測：

```txt
同分類任務是否有相似完成率？
同分類任務是否有相似跳過率？
同分類任務是否有相似最佳時段？
同分類任務是否有相似耗時偏差？
同分類任務是否對重要任務完成度有一致影響？
```

若分類太寬泛：

```txt
某分類內部表現差異大，建議拆分。
```

若分類預設不準：

```txt
某分類實際耗時長期高於預估，建議調整預設時間。
某分類重要程度與實際行為不一致，建議重新確認重要程度。
```

### 5.7 PCA 與長期偏好模型

PCA 不作為 v0.1 核心推薦依據，但可在 v0.2 後用於長期行為分析。

可用 feature vector：

```txt
importance
urgency
energy
taskType
startHour
weekday
estimateMinutes
actualMinutes
wasSkipped
wasCompleted
previousSessionType
dailyLoadRatio
completedBeforeStart
```

用途：

```txt
找出影響完成度的主要維度
壓縮多個行為特徵
辨識長期工作偏好
偵測工作型態群集
```

產品上不直接顯示 PCA 術語，而轉為可讀建議：

```txt
最近完成度主要受開始時段和任務能量需求影響。
高能量任務若安排在上午，完成率較高。
```

### 5.8 優化組回饋前期規劃組

優化組產生的建議應回到前期規劃組，由使用者決定是否更新分類預設或工作流偏好。

流程：

```txt
使用者建立任務
-> 套用分類預設
-> 工作組執行
-> 優化組分析
-> 發現分類預設偏差或矩陣失衡
-> 產生調整建議
-> 使用者確認
-> 回寫分類預設與工作流設定
```

可調整項：

```txt
分類預設重要程度
分類預設能量
分類預設估時
分類建議時段
矩陣目標配比
工作流位置偏好
推薦關係權重
```

## 6. AI 組

### 6.1 功能敘述

AI 可橫跨三個主要功能組，但 v0.1 不應依賴 AI 完成核心流程。

可支援：

```txt
前期規劃：
  拆解快速卸載文字
  推測分類、重要程度、期限、能量與大小
  把模糊任務改寫成下一步

工作：
  任務卡住時提出下一個 5 分鐘動作
  解釋推薦原因
  將跳過行為轉成替代方案

優化：
  解釋矩陣失衡原因
  解釋分類有效性
  生成週回顧
  提出可確認的關係描述假設
```

### 6.2 AI 連接方式

不要在前端直接存放 API key。

未來架構：

```txt
PWA
-> HTTPS request
-> Cloudflare Worker / Supabase Edge Function
-> AI Provider
-> 回傳結構化建議
```

AI 請求只傳必要摘要：

```ts
type AdvisorPayload = {
  tasksSummary: TaskSummary[];
  workloadSummary: WorkloadSummary;
  recentStats: DailyStats[];
  detectedRelations?: WorkflowRelation[];
  userQuestion?: string;
};
```

AI 建議不得直接改資料，需轉成待確認建議：

```txt
使用者確認後，才更新分類預設、推薦關係或矩陣目標。
```

## 8. 底層資料流

```txt
前期規劃輸入
-> TaskTypeProfile 套用預設
-> Task Repository
-> Workload Engine
-> Recommendation Engine
-> Focus Screen
-> Timer Session
-> Work Events
-> Analytics / Optimization
-> Workflow Relations / Profile Suggestions
-> User Confirmation
-> 回寫規劃預設與推薦排序
```

核心原則：

```txt
任務模型保持乾淨。
特殊現象不新增任務欄位。
長期效果透過事件資料與關係描述推論。
使用者保有最終決定權。
```

```txt
專案可啟動

核心型別已建立
```

### M6：進階關係偵測

目標：透過簡單演算法偵測促進與抑制關係。

內容：


## 12. 核心產品原則

```txt
不是管理任務，而是把今天變得可開始。

不是追求更多欄位，而是保留乾淨模型，用事件和關係描述解釋長期效果。

不是只追 deadline，而是同時保護重要不緊急任務。

不是讓 AI 取代使用者，而是讓 AI 幫助使用者理解自己的工作模式。

不是一次做到完美，而是透過規劃、執行、記錄、分析、校正形成閉環。
```
